回流与重绘

讲清楚重排或回流、重绘

为了让项目的运行效率更高，减少开销，需要对重排、回流、重绘有一定的了解。 重排和回流其实是一回事。 浏览器在渲染页面的时候，大致是以下几个步骤： 解析html生成DOM树，解析css，生成CSSOM树，将DOM树和CSSOM树结合，生成渲染树； 根据渲染树，浏览器可以计算出网页中有哪些节点，各节点的CSS以及从属关系 - 回流 根据渲染树以及回流得到的节点信息，计算出每个节点在屏幕中的位置 -

https://zhuanlan.zhihu.com/p/342371522

回流重绘队列

浏览器会维护一个队列,把所有引起回流和重绘的操作放入队列中,如果队列中的任务数量或者时间间隔达到一个阈值的,浏览器就会将队列清空,进行一次批处理,这样可以把多次回流和重绘变成一次。

强制回流

当你访问以下属性或方法时,浏览器会立刻清空队列

clientWidth、clientHeight、clientTop、clientLeft
offsetWidth、offsetHeight、offsetTop、offsetLeft
scrollWidth、scrollHeight、scrollTop、scrollLeft
width、height
getComputedStyle()
getBoundingClientRect()

为队列中可能会有影响到这些属性或方法返回值的操作,即使你希望获取的信息与队列中操作引发的改变无关,浏览器也会强行清空队列,确保你拿到的值是最精确的。

避免回流

  • 使用transform而不用top，botom等修改
  • 将要修改的元素display设置为none，对其一顿修改再恢复display，这样只触发两次回流
  • DocumentFragment，文档片段接口，一个没有父对象的最小文档对象。它被作为一个轻量版的 Document 使用，就像标准的 document 一样，存储由节点（nodes）组成的文档结构。与 document 相比，最大的区别是 DocumentFragment 不是真实 DOM 树的一部分，它的变化不会触发 DOM 树的重新渲染，且不会导致性能等问题。

渲染层合并 (Composite)

对于页面中 DOM 元素的绘制(Paint)是在多个层上进行的。在每个层上完成绘制过程之后,浏览器会将绘制的位图发送给 GPU 进程绘制到屏幕上,将所有层按照合理的顺序合并成一个图层,然后在屏幕上呈现。

webkit浏览器内核（渲染进程）渲染过程

从 8 道面试题看浏览器渲染过程与性能优化

  • 渲染进程又称为浏览器内核
  • GUI 渲染线程与 JS 引擎线程是互斥的
  • CSS加载与阻塞
    • CSS 加载不会阻塞 DOM 的解析（CSSOM和DOM并行构建）
    • CSS加载会阻塞DOM渲染（只有当CSSOM和DOM同时解析完毕后，才会生成渲染树）
      • 出于性能考虑，如果先渲染dom，css修改样式后还得触发回流和重绘
      • 避免用户看到长时间白屏，必须提高css加载速度
    • css 加载会阻塞后面 js 的执行，js阻塞DOM解析（线程互斥）
  • DOMContentLoaded事件与window.onload的区别
    • 为什么一再强调将css放在头部，将js文件放在尾部

为什么一再强调将css放在头部，将js文件放在尾部

  • 现在的浏览器为了用户体验，要减少白屏时间，所以就一边解析html文档内容生成DOM树，一边将已经解析好的部分DOM和CSSOM渲染到页面上。
  • 将css放在头部，是为了并行解析DOM和CSSOM，让页面能先渲染尽可能多的内容
  • js放在尾部，是因为如果js放在头部，js引擎会解析执行js代码，GUI渲染线程和js引擎线程互斥，会阻碍DOM的解析，增加白屏时间
  • 但是DOMContentLoaded事件触发的时间是一样的，因为只有解析了整个HTML文档后才触发，js和css在哪都一样

提高css加载速度

  1. 使用cdn
  2. webpack打包或者gzip压缩
  3. 使用缓存（cache-control，expires，E-tag，缓存文件名后加版本号区分新旧版本）
  4. 减少http请求数，将多个css文件合并
